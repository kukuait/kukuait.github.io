# GCP Cloud Monitoring 的 Uptime Checks 與 Alerting 功能

> Uptime checks 是「從外部主動探測服務是否還活著」的黑箱監控；Alerting 是「當任何指標（包含 uptime check 的結果）出現異常時通知人」的規則引擎 —— 兩者常搭配使用，但分別解決不同層次的問題。

## Step 1：Uptime Checks 解決什麼問題

Cloud Monitoring 內建的 metrics（CPU、延遲、錯誤率⋯）多數是**服務自己回報**的內部視角：如果整台機器掛掉、DNS 解析失敗、或網路層被防火牆擋住，服務可能完全來不及回報任何 metric。

Uptime check 是**黑箱探測（black-box probing）**：GCP 從全球多個地點主動對你指定的 endpoint 發送請求，只關心「探測結果」，不需要目標服務有任何內建的監控程式碼。

- **協定**：HTTP（S）、TCP
- **探測地點**：可選單一地點或多個全球地點（美、歐、亞太⋯），每個地點各自獨立判定成功 / 失敗
- **檢查頻率**：1、5、10、15 分鐘
- **內容驗證**：不只看 status code，還能設定 response body 必須（或不能）包含某段文字，抓「回了 200 但頁面其實是錯誤訊息」這種偽陽性
- **Private uptime check**：透過 VPC 與 PSC（Private Service Connect）可探測沒有對外 IP 的內部服務，不需要為了健康檢查而把服務暴露到公網

探測結果會寫成標準的 metric 時間序列，最關鍵的是：

```text
monitoring.googleapis.com/uptime_check/check_passed   # 0/1，本次探測是否成功
monitoring.googleapis.com/uptime_check/request_latency # 探測的往返延遲
```

這代表 uptime check 產生的資料**跟其他 metric 完全同構**—— 可以畫進 dashboard（見 [Cloud Monitoring Dashboard 與 Metrics Explorer 的關係](#/sre/05-gcp/gcp-monitoring-dashboard.mdx)），也可以直接接上 alerting policy。

## Step 2：Alerting 解決什麼問題

Alerting policy 是一個獨立於 uptime check 的通用規則引擎，輸入可以是**任何** Cloud Monitoring 的 metric（uptime check、CPU 使用率、自訂的 application metric、log-based metric⋯），核心組成：

| 組成 | 說明 |
|---|---|
| **Condition** | 觸發規則，例如「metric 超過門檻值」「metric 變化率異常」「metric 完全沒有回報數據（absence）」 |
| **Aggregation** | 如何彙總多個時間序列，例如「99th percentile」「跨全部 uptime check 地點取平均通過率」 |
| **Notification channel** | 通知去哪：Email、Slack、PagerDuty、SMS、Pub/Sub、Webhook、Mobile app push |
| **Documentation** | 附在通知裡的說明文字，通常放 runbook 連結，讓值班的人不用先自己 google |
| **Incident 生命週期** | 條件持續滿足 → open incident；條件恢復正常 → 自動 close（也可設定 auto-close 逾時） |

Alerting policy 的條件判定不是「單點觸發」，而是要求**持續滿足一段時間（duration）**才 fire，避免單次抖動誤報。這對 uptime check 特別重要 —— 單一地點的一次探測失敗可能只是那個 probing location 本身網路波動，所以典型作法是設定「多個地點中有超過 N 個回報失敗」或「所有地點的通過率低於 X%」才觸發。

## Step 3：兩者如何搭配

```mermaid
flowchart LR
    A["Uptime Check<br/>(多地點主動探測)"] -->|"寫入"| B["uptime_check/check_passed<br/>metric 時間序列"]
    C["應用程式 / 基礎設施<br/>(CPU、延遲、自訂 metric)"] -->|"回報"| D["其他 metric<br/>時間序列"]
    B --> E["Alerting Policy<br/>(condition + aggregation)"]
    D --> E
    E -->|"條件持續滿足"| F["Incident 開啟"]
    F --> G["Notification Channel<br/>(Slack / PagerDuty / Email...)"]
    F -->|"條件恢復"| H["Incident 自動關閉"]
```

常見的落地模式：

1. **服務可用性告警**：對 uptime check 的 `check_passed` 建立 alerting policy，條件是「跨全部探測地點的失敗比例 > 50% 且持續 5 分鐘」，通知 on-call。這是偵測「服務完全打不通」最外部、最貼近真實用戶視角的一層，不依賴服務本身還能不能回報 metric。
2. **延遲告警**：對 `uptime_check/request_latency` 設門檻值，抓「服務有回應但變慢」。
3. **SLO burn rate 告警**：如果已經定義了 SLO（見可靠性基礎章節），可以用 uptime check 或其他 metric 的歷史資料算 error budget 消耗速度，對「燒錢太快」本身建立告警，而不是對單次錯誤告警 —— 這樣可以區分「偶發雜訊」和「真的在快速惡化」。
4. **Absence 告警**：條件設成「這個 metric 超過 N 分鐘沒有任何資料點」，用來抓 uptime check 探測本身掛掉、或服務完全停止回報（比一般的門檻值告警更早抓到「服務整個消失」的情況）。

## Step 4：Uptime Check 本身怎麼「送」通知 —— 關鍵是它不會自己送

一個常見誤解是「幫 uptime check 打開通知選項」。實際上 **uptime check 資源本身沒有通知能力**，它只負責持續寫入 `check_passed` / `request_latency` 這兩條 metric 時間序列；會不會通知人、通知誰，完全是掛在它上面的 **Alerting Policy** 決定的。

```mermaid
sequenceDiagram
    participant P as "Prober（多地點探測)"
    participant M as "Cloud Monitoring<br/>(metric 時間序列)"
    participant A as "Alerting Policy<br/>(condition 引擎)"
    participant I as "Incident"
    participant N as "Notification Channel"
    participant R as "值班人員"

    loop 每次檢查週期
        P->>M: 寫入 check_passed(0/1)
    end
    M->>A: condition 持續評估(streak/fraction)
    A->>A: 判定「失敗地點比例 > 門檻」且「持續時間達標」
    A->>I: 條件滿足 → open incident
    I->>N: 依 policy 綁定的 channel 逐一發送
    N->>R: Email / Slack / PagerDuty / SMS / Pub-Sub / App push
    Note over A,I: 條件恢復正常 → incident 自動 close，<br/>也會依 policy 設定發「已恢復」通知
```

實際落地時分兩段設定，缺一不可：

1. **建立 Alerting Policy 的 condition**：在 GCP Console 的 Uptime Checks 頁面點「Create Alerting Policy」時，會自動帶出一個預填的 condition 樣板（metric 固定為該 uptime check 的 `check_passed`），但你仍要選：
   - **判定範圍**：`any region reports failures`（任一探測地點失敗就算）還是 `more than X% of regions report failures`（多數地點都失敗才算）—— 前者靈敏、容易因單地點網路波動誤報，後者較穩但反應慢。
- **持續時間（duration）**：條件要連續滿足多久才觸發，例如「持續 5 分鐘」。
2. **綁定 Notification Channel**：Notification channel 是獨立資源（在 Monitoring → Alerting → Notification channels 底下建立與管理），同一個 channel 可以被多個不同的 alerting policy 重複使用。常見型別與送達方式：

| Channel 型別 | 實際送達機制 |
|---|---|
| Email | SMTP 寄送到指定信箱 |
| SMS | 簡訊閘道寄送到指定號碼（需先驗證） |
| Slack | 透過 Slack App 整合，以 webhook 形式貼文到指定頻道 |
| PagerDuty | 呼叫 PagerDuty 的 Events API，直接開一張 PagerDuty incident |
| Webhook | 對你指定的 URL 發 HTTP POST，body 帶 incident JSON，可接任何自架系統 |
| Pub/Sub | 發一則訊息到指定 topic，適合接自動化處理（例如自動觸發 runbook 腳本） |
| Mobile App | 透過 Google Cloud Console App 推播 |

沒有第 2 步，即使 alerting policy 的 condition 正確觸發、incident 也確實開了，也不會有任何人收到通知 —— 這是排查「uptime check 明明掉了但沒人被吵醒」時第一個要檢查的地方：**policy 有沒有掛 notification channel、channel 本身有沒有驗證通過（例如 email/SMS 需要先完成一次驗證流程才會生效）**。

## Step 5：Uptime Check 能不能只在特定時段檢測

答案是**不行**——uptime check 資源沒有時間排程欄位,只要建立了就會依設定的頻率全年無休地探測。想要「只在 09:00~18:00 檢測 `/healthz`」這種效果,實際上要控制的是**告警要不要通知人**,而不是探測本身,有兩種做法:

1. **排程開關 Alerting Policy**：uptime check 照常 24 小時探測,改用 Cloud Scheduler 在時段起訖點呼叫 Monitoring API,把對應 alerting policy 的 `enabled` 欄位在 09:00 設 `true`、18:00 設 `false`。policy disabled 期間即使探測失敗也不會開 incident、不會發通知。

   ```bash
   # 09:00 觸發
   gcloud alpha monitoring policies update POLICY_ID --enabled
   # 18:00 觸發
   gcloud alpha monitoring policies update POLICY_ID --no-enabled
   ```

2. **通知層過濾**：policy 保持一直 enabled,但 notification channel 不直接接 Slack/PagerDuty,而是先接 Pub/Sub → 一個小的 Cloud Function,由 function 檢查目前時間是否落在時段內,是才轉發給真正的通知管道。適合「還想保留完整 incident 歷史做事後分析,只是不想在非時段吵人」的情境。

| | 排程 enable/disable policy | 通知層過濾 |
|---|---|---|
| 實作複雜度 | 低(兩個 Cloud Scheduler job + API 呼叫) | 中(要維護一個轉發 function) |
| Incident 記錄 | 非時段內完全不會產生 incident | 非時段內仍會產生 incident,只是不轉發通知 |
| 適用情境 | 只想控制「什麼時候吵人」 | 還想保留完整 incident 歷史 |

如果 `/healthz` 在非時段本來就會回應失敗(例如服務本身會 scale to zero),問題其實是這個 endpoint 的語意不適合用一般 uptime check 去理解——通常直接接受「該時段本就預期失敗」,靠上述兩種做法把告警噪音關掉,而不是嘗試讓探測本身暫停(GCP 沒有提供這個開關)。

## Step 6：Snooze 與 Incident 的差異

Incident（見 Step 2–4）是「alerting policy 的 condition 從滿足到恢復」這段期間的紀錄本身：policy 每觸發一次就生一個新的 incident，結束方式是條件恢復後自動 close，或值班人員手動 acknowledge/close。

Snooze 是另一個獨立資源：在指定的時間窗口內，暫停某些 alerting policy 觸發時的**通知**，但不影響 incident 是否照常產生。一句話比較：

**Incident 回答「發生了什麼」；Snooze 回答「這段時間要不要吵醒人」。**

|  | Incident | Snooze |
|---|---|---|
| 本質 | Alerting Policy 觸發後產生的事件紀錄 | 一段時間窗口內、對指定 policy 的通知抑制規則 |
| 誰觸發 | Condition 滿足時系統自動建立 | 人工手動建立（例如已知維護窗口、正在處理同一個根因、告警太吵想先靜音再調參數） |
| 對歷史紀錄的影響 | 本身就是歷史紀錄 | 不影響 incident 是否產生，只抑制新增的通知 |
| 生命週期 | Condition 恢復 → 自動 close，或手動 acknowledge/close | 到期時間一到自動失效，也可手動提前結束 |
| 適用情境 | 追蹤「這次異常從什麼時候到什麼時候」 | 「我知道現在會觸發，不需要再通知我」 |

跟 Step 5 提到的時段控制通知相比，Snooze 是 GCP 原生提供的做法：直接在 Alerting 頁面建立一個 Snooze，指定要 mute 的 policy 與起訖時間即可，不用像 Step 5 方案二那樣自己搭 Pub/Sub + Cloud Function 做轉發過濾。而且 Snooze 期間 incident 仍照常產生、歷史紀錄完整，這點跟 Step 5 方案一「直接把 policy disable」不同——disable 期間連 incident 都不會生，事後無法回頭確認那段時間到底觸發過幾次；Snooze 則是「事件照樣記錄，只是不吵人」，更適合事後要做 postmortem 或計算 error budget 消耗的場景。

## Step 7：Uptime Check 能不能測試需要 OIDC Token 的 endpoint

能，但僅限於「目標端點驗證的是 Google 簽發的 OIDC identity token」這一種情境，靠的是 Uptime Check 設定裡的 **Service Agent Authentication** 選項——這個選項只在 HTTPS check 才看得到。

啟用後，每次探測前 Cloud Monitoring 會自動幫一個專屬的 monitoring service agent 產生一張 Google 簽發的 identity token：

```text
service agent：service-PROJECT_NUMBER@gcp-sa-monitoring-notification.iam.gserviceaccount.com
token issuer （iss）：https://accounts.google.com
token audience （aud）：自動等於你填的目標 URL
```

這張 token 會以 `Authorization: Bearer <token>` header 的形式跟著探測請求一起送出。你的服務端驗證方式跟一般 Google OIDC token 完全一樣——用 `google-auth` 套件的 `id_token.verify_oauth2_token()` 檢查簽章、`iss`、`aud` 即可，這正是〔[GCP Alerting Policy → Pub/Sub → 自訂 Webhook 完整串接](#/sre/99-staging/gcp-alerting-pubsub-webhook.mdx)〕裡驗證 Pub/Sub push OIDC token 的同一套手法。

如果目標是 Cloud Run 或 Cloud Functions（gen2）這種**平台層本身就用 IAM 做 invoker 檢查**的服務，光是應用程式碼會驗證 token 還不夠，必須另外把 `roles/run.invoker` 授權給這個 monitoring service agent 的 email，否則請求在平台層就被擋下 403，根本進不到你的程式碼。

### 各種驗證需求對照表

| 目標端點的驗證需求 | Uptime Check 能不能測 | 怎麼設定 |
|---|---|---|
| 公開、無驗證 | 可以 | 預設 HTTP/HTTPS check |
| 固定的 API key／static Bearer token | 可以 | Custom headers（可加密隱藏值） |
| Basic Auth（帳號密碼） | 可以 | Authentication 欄位選 Basic |
| 需要 Google 簽發、audience 等於目標 URL 的 OIDC token（如 Cloud Run IAM 保護） | 可以 | Authentication 欄位選 Service Agent Authentication，僅 HTTPS |
| 第三方 OIDC provider 動態核發、audience 需自訂的 token | 不行 | 沒有原生支援，見下方 |

### 這個機制的限制

- **身份固定**：只能用這個固定的 monitoring service agent，不能像 Cloud Scheduler 的 OIDC 設定那樣指定「用哪個自訂 service account 產生 token」。
- **audience 不可自訂**：`aud` 固定等於你填的目標 URL，如果驗證邏輯要求別的 audience 值，就永遠對不上。
- **跟 custom header 互斥**：Authentication 欄位設了 Service Agent Authentication，就不能再自己手動加 `Authorization` header，兩者同時設定會直接噴 `Invalid Argument` 錯誤。
- **TCP check 不支援**：這個選項只存在於 HTTPS check。

如果你的 endpoint 要驗證的是第三方 OIDC provider（非 Google）核發、且 audience 需要自訂的 token，Uptime Check 本身無法產生這種 token，只能退而求其次：改用 **Synthetic Monitor**（本質是讓你自己寫一段 Cloud Function 當探測邏輯），在 function 裡自己走完整套換票流程再打目標 endpoint，缺點是要多維護一段程式碼，換來的是驗證流程完全不受限於 Uptime Check 內建的這幾種選項。

### 主控台看不到 Service Agent Authentication 選項時，先檢查這兩件事

這個選項**只在 Protocol 設為 HTTPS 時才會出現**，是排查「Console 建立畫面的 Authentication 沒有這個選項」時第一個要確認的地方：

1. **Protocol 選錯**：Uptime check 的 target 步驟裡，Protocol 選的是 HTTP 或 TCP，Authentication 這個子區塊本身就不會顯示 Service Agent Authentication；TCP 甚至整個 Authentication 區塊都不會出現。把 Protocol 切回 HTTPS 再看一次。
2. **看錯步驟**：Authentication 是在填完 hostname／path 之後、同一頁的「更多 target 選項（More target options）」裡才會展開，不是獨立的頁籤，容易被忽略而以為選項不存在。

如果 Protocol 已經是 HTTPS、也確認展開了 more target options，Console 裡卻還是只看到 Basic Authentication，可以直接改用 gcloud 或 Terraform 建立／更新這個 uptime check config，兩者對應的欄位是 `httpCheck.serviceAgentAuthentication.type = OIDC_TOKEN`：

```hcl
resource "google_monitoring_uptime_check_config" "healthz" {
  display_name = "healthz-oidc"
  timeout      = "10s"

  http_check {
    path         = "/healthz"
    port         = "443"
    use_ssl      = true
    validate_ssl = true

    service_agent_authentication {
      type = "OIDC_TOKEN"
    }
  }

  monitored_resource {
    type = "uptime_url"
    labels = {
      project_id = "my-project"
      host       = "example.com"
    }
  }
}
```

若目標是 Cloud Run，記得同時把 `roles/run.invoker` 授權給 monitoring service agent（`service-PROJECT_NUMBER@gcp-sa-monitoring-notification.iam.gserviceaccount.com`），這一步 Console／Terraform 都不會幫你自動做：

```bash
gcloud run services add-iam-policy-binding my-service \
  --region=asia-east1 \
  --member="serviceAccount:service-PROJECT_NUMBER@gcp-sa-monitoring-notification.iam.gserviceaccount.com" \
  --role="roles/run.invoker"
```

## 兩者的本質差異

| | Uptime Check | Alerting |
|---|---|---|
| **角色** | 資料來源（黑箱探測器） | 規則引擎（對任意資料源做判斷與通知） |
| **視角** | 外部、跨地理位置的真實用戶視角 | 中立，可以接內部或外部任何 metric |
| **能否獨立存在** | 可以只建 uptime check 不建告警（純觀測） | 可以不靠 uptime check，對任何既有 metric 告警 |
| **典型用途** | 偵測「服務對外是否可達」 | 把「異常」轉成「通知人」的最後一哩路 |
