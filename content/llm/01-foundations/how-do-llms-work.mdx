# LLM 的運作原理

> 一句話版本：LLM 是一個用巨量文字訓練出來的「下一個 token 預測器」—— 它把輸入文字切成 token、轉成向量、經過幾十層 Transformer 運算後，輸出「下一個 token 是什麼」的機率分佈，然後把選出的 token 接回輸入、再預測下一個，如此反覆直到生成結束。

## Step 1 —— Tokenization：把文字切成 token

模型不直接處理文字，而是處理 **token**（字塊）。Tokenizer 依照訓練好的詞彙表（vocabulary，通常 3 萬～25 萬個）把輸入切開：

```
"今天天氣真好" → ["今天", "天氣", "真", "好"]     (示意)
"unbelievable" → ["un", "believ", "able"]        (示意)
```

- 常見演算法是 BPE（Byte Pair Encoding）：從單一字元開始，反覆把最常一起出現的相鄰片段合併成新 token。
- 每個 token 對應一個整數 ID，所以模型看到的其實是一串整數，例如 `[15496, 995, 11, ...]`。
- 這也解釋了一些 LLM 的怪癖：數字算術不穩、算不清單字裡有幾個字母 —— 因為模型看到的是 token，不是逐字元的拼寫。

## Step 2 —— Embedding：把 token 變成向量

每個 token ID 透過一張查表（embedding matrix）換成一個高維向量（例如 4096 維）。這個向量是訓練出來的，語意相近的 token 在向量空間中會靠得比較近。到這一步，一句話變成了一個「向量序列」。

## Step 3 —— Transformer 層層加工：attention 讓 token 互相看見

向量序列接著流過幾十層（大模型可到上百層）結構相同的 **Transformer decoder layer**，每層做兩件事：

1. **Self-attention**：每個 token 對前面所有 token 計算「相關性分數」，然後按分數加權混合彼此的資訊。這就是模型理解上下文的機制 —— 例如「它」這個 token 可以透過 attention 去「看」前文中的「那隻貓」，把指代關係吸收進自己的向量裡。
2. **Feed-forward network（FFN）**：對每個 token 的向量各自做一次非線性轉換，一般認為這裡儲存了大量「知識」。

每經過一層，每個位置的向量就融合了更多上下文與知識，變得越來越「懂」這句話。細節可見〈什麼是 Transformer？〉一篇。

## Step 4 —— 輸出：一個機率分佈，然後抽樣

走完所有層之後，取**最後一個位置**的向量，乘上一個輸出矩陣、經過 softmax，得到「詞彙表中每個 token 是下一個 token」的機率分佈：

```
輸入: "台灣最高的山是"
輸出機率: { "玉山": 0.92, "雪山": 0.03, "阿里山": 0.01, ... }
```

接著依取樣策略選出一個 token：

| 策略 | 行為 |
|------|------|
| greedy | 永遠選機率最高的，輸出固定但容易呆板 |
| temperature | 調整分佈的「尖銳程度」，越高越隨機、越有創意 |
| top-p /top-k | 只在機率累積前 p（或前 k 名）的候選中抽樣，砍掉長尾 |

這就是為什麼同一個 prompt 每次回答可能不一樣 —— 生成本質上是**抽樣**。

## Step 5 —— 自回歸迴圈：一次一個 token，直到結束

把剛選出的 token 接回輸入序列尾端，整個流程再跑一次，預測下一個 token…… 如此反覆，直到模型產出特殊的結束 token（如 `<|eot|>`）或達到長度上限。這種「用自己的輸出當下一步的輸入」的方式稱為 **autoregressive（自回歸）生成**。

工程上的推論：生成 1000 個 token 就要跑 1000 次前向運算，所以推理成本與輸出長度成正比；為了不每次重算前文的 attention，實務上會用 KV cache 快取中間結果（屬於推理優化主題）。

## Step 6 —— 這些能力是怎麼訓練出來的？

上面講的是「推理時」的運作；模型參數本身來自三個階段的訓練：

1. **Pretraining（預訓練）**：拿數兆 token 的網路文字，玩同一個遊戲 ——「遮住下一個 token，讓模型猜」。猜錯就用梯度下降微調參數。規模夠大之後，為了把「下一個 token」猜準，模型被迫學會文法、事實知識、甚至推理模式。這一步產出的是 base model，只會接龍，不會對話。
2. **SFT（Supervised Fine-Tuning）**：用人工撰寫的「指令 → 理想回答」範例繼續訓練，教會模型「用助理的口吻回答問題」這個格式。
3. **RLHF / DPO（對齊）**：讓人類（或 reward model）比較模型的多個回答哪個更好，據此調整模型偏好，使輸出更有幫助、更安全。

## 一個常見誤解：LLM 不是資料庫查詢

LLM 沒有在「檢索」一份儲存好的答案；知識以分散的形式壓縮在數千億個參數（權重）裡，回答是逐 token「算」出來的。這解釋了兩件事：

- **為什麼會 hallucination**：模型永遠能給出一個「機率上通順」的下一個 token，即使它對應的事實不存在。
- **為什麼有知識截止日**：參數在訓練完成後就固定了，之後的新知識要靠 RAG、工具呼叫等外掛機制補上。
