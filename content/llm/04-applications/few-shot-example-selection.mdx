# Few-shot 範例選擇的最佳實踐

> 一句話版本：Few-shot 的品質**完全取決於範例怎麼挑**——格式不一致、類別覆蓋不全、分佈偏離真實輸入，三個陷阱任何一個都會讓 few-shot 比 zero-shot 還差。

## Step 1：為什麼範例挑選這麼重要？

Few-shot 觸發的是 In-context Learning（ICL）——模型從範例中**歸納格式**，而不是靠你用文字解釋。這代表範例就是訓練信號。爛的訓練信號教出爛行為。

## Step 2：五個選例原則

### 原則 1：格式絕對一致

每組範例的輸入/輸出結構要一模一樣：分隔符、欄位順序、換行方式全部統一。

```text
❌ 差：格式不一致
輸入：這電影真好看
標籤：正面情感

輸入：很無聊
結果 → 負面

輸入：還可以
→ 中性

✅ 好：格式統一
輸入：這電影真好看  標籤：正面
輸入：很無聊        標籤：負面
輸入：還可以        標籤：中性
```

格式不一致時，模型看到的是「三種不同任務」，ICL 完全失效。

### 原則 2：覆蓋所有輸出類別（label coverage）

涵蓋所有可能的輸出種類，尤其是邊界案例。如果情感分類有「正面/負面/中性」三類，三類都要有範例——只給正面和負面，模型幾乎不會輸出中性。

**特別注意容易混淆的邊界**：「諷刺」語氣（字面正面、語意負面）、程度詞（「還不錯」到底算正面還是中性？）——這些案例放進範例比任何文字說明都有效。

### 原則 3：分佈接近真實輸入

範例的語氣、長度、風格要和你實際會傳進去的資料相似。

| 場景 | 範例應來自 |
|------|-----------|
| 電商評論情感分析 | 電商評論，非新聞或文學 |
| 合約條款抽取 | 真實合約格式，非虛構範例 |
| 程式碼審查 | 同語言同風格的 code |

範例和真實輸入的分佈差距越大，ICL 的泛化效果越差。

### 原則 4：3~8 個通常夠用

超過 8 個範例，邊際效益快速遞減，還白白佔用 context window。更多範例 ≠ 更好。

| 範例數 | 建議 |
|--------|------|
| 1~2 | 通常不夠覆蓋所有類別 |
| 3~5 | 多數任務的甜蜜點 |
| 6~8 | 格式複雜或類別多時適用 |
| 8 以上 | 考慮改用 fine-tuning |

### 原則 5：最強範例放後面（Recency bias）

模型對 prompt 末尾的範例更敏感（Recency bias）。把**最有代表性**、**最難的**案例放在最後幾個，讓模型「最後記住」的格式是你最想要的。

## Step 3：選例 Checklist

| 檢查項 | 說明 |
|--------|------|
| ✅ 格式完全統一 | 分隔符、冒號、換行方式一致 |
| ✅ 所有輸出類別都有範例 | 不能遺漏任何一種輸出 |
| ✅ 覆蓋邊界/困難案例 | 容易混淆的情況最需要範例 |
| ✅ 分佈接近真實輸入 | 風格、長度、語言要和真實輸入相近 |
| ✅ 3~8 個範例 | 超過效益遞減，不足覆蓋不夠 |
| ✅ 最強範例放後面 | 利用 recency bias |
