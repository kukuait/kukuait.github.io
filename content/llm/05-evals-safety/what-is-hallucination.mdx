# LLM 的 Hallucination 問題

> LLM 生成了**聽起來合理但實際上是錯誤或虛構**的內容，且通常表現得極為自信。

---

## 核心原因：LLM 是語言模型，不是事實資料庫

LLM 的本質是一個**機率語言模型**：給定前面的 token，預測下一個 token 的機率分佈。它學到的是「哪些詞彙序列在訓練資料中共同出現」，而不是「什麼是客觀事實」。

```mermaid
flowchart LR
    A["輸入 prompt"] --> B["Transformer<br/>計算 token 機率"]
    B --> C["取樣下一個 token"]
    C --> D["再計算下一個..."]
    D --> E["輸出文字"]

    style B fill:#3b5bdb,color:#fff
    style E fill:#2f9e44,color:#fff
```

這個機制帶來幾個根本性問題：

1. **訓練資料的幻覺**：訓練資料本身可能包含錯誤、過時或矛盾的資訊。
2. **知識截止日期**：模型對訓練截止日後的事件一無所知，被問到時仍可能「猜」。
3. **模式泛化的副作用**：模型學會了「引用學術論文」的語言模式，但不記得具體哪篇說了什麼，於是生成格式正確但不存在的引用。
4. **RLHF 的副作用**：訓練傾向讓模型給出「看起來有用」的答案，可能強化了自信地說出錯誤答案的傾向。

---

## Hallucination 的常見類型

| 類型 | 說明 | 例子 |
|------|------|------|
| **事實性錯誤** | 斷言了錯誤的事實 | 「愛因斯坦因相對論得諾貝爾獎」（實際上是光電效應）|
| **幻覺引用** | 虛構不存在的論文/書籍 | 生成看似真實的 DOI、作者、期刊名，但查無此文 |
| **幻覺 API** | 編造不存在的函式或參數 | 虛構 library 的方法名稱與簽名 |
| **細節遺失** | 概念正確，但細節錯誤 | 事件發生年份、人名拼寫等小細節 |
| **上下文矛盾** | 在同一對話中前後矛盾 | 先說 A 是對的，後說 A 是錯的 |

## 如何緩解？

### 使用者／工程師可控制的層面

- **RAG（Retrieval-Augmented Generation）**：把事實文件放進 context，要求模型只從文件中回答，讓回答「有所依據」。
- **要求引用來源**：「請引用你答案的依據」——模型若找不到來源會更謹慎。
- **降低 temperature**：減少隨機性，讓模型傾向更保守的高機率輸出。
- **明確允許「不知道」**：Prompt 中加入「如果你不確定，請直接說不確定」。

### 模型廠商在做的事

- **RLHF / Constitutional AI**：用人類或 AI 反饋訓練模型，在不確定時拒絕回答。
- **自我一致性（Self-consistency）**：同一問題多次取樣，取多數決結果，降低隨機錯誤。
- **Grounding 架構**：官方整合搜尋引擎或知識庫，讓模型回答前先檢索。

---

## 工程師的實務心法

Hallucination 無法被「消滅」，只能被「管理」。

任何需要**高準確度**的場景（醫療、法律、財務、程式碼生成），建議：

1. 搭配 RAG，讓模型有可查核的資料來源。
2. 建立 eval pipeline，定期測試模型在關鍵問題上的準確率。
3. 設計 UI 讓使用者清楚知道「這是 AI 生成內容，請自行驗證」。
